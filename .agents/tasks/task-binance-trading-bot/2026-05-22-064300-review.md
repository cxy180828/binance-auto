# Binance Alpha momentum scalper

A greenfield automated trading bot targeting Binance Alpha market tokens. It watches 1-minute candles for a 5%+ price spike with elevated volume, enters a market buy, then manages the position with a trailing stop (activated at +5%, triggers on a 2% drop from peak) or a time-based stop-loss (exits if the trailing stop hasn't activated within 3 minutes). A blacklist mechanism suppresses symbols after consecutive losses, and a Feishu webhook delivers trade alerts and a daily summary.

The architecture separates concerns well: exchange wrapper, strategy engine, blacklist manager, notifier, and SQLite storage are injected as dependencies, making the strategy fully testable via mocks. The test suite covers signal detection, trailing stop mechanics, time stop-loss, cooldowns, daily limits, and blacklist behavior.

**Watch for:** In-memory position state is lost on crash with no mechanism to reconcile against exchange balances (confirmed). The trailing stop price is recalculated every tick but never ratchets upward with a `max()` guard — correctness depends on `highest_price` never being reset (confirmed). The blacklist manager's in-memory loss counter diverges from the database's `get_consecutive_losses` query, which is dead code (confirmed). API credentials are stored in plain YAML with no `.gitignore` protection (confirmed).

## High-level view

The signal detection uses the last closed candle (index -2) and compares volume against a 5-candle rolling average from indices -7 to -3. The guard `len(klines) < 7` prevents index errors. The volume window excludes the signal candle itself, avoiding look-ahead bias.

The trailing stop overwrites `trailing_stop_price` every tick rather than applying `max(existing, new)`. Since `highest_price` only ratchets upward, the stop price also only increases in practice — but this invariant is implicit, not enforced. If `highest_price` tracking ever had a reset path, the stop could move downward.

Position state lives entirely in `self.positions` with no persistence. A crash leaves purchased tokens unmanaged in the exchange wallet with no recovery mechanism on restart.

The blacklist manager and `Storage.get_consecutive_losses` are parallel implementations of the same concept that never communicate. After a restart the blacklist is always empty, regardless of recent loss history in the database.

The bot scans all USDT pairs every loop iteration. On mainnet with 600+ pairs, this exhausts rate limits within a single pass. No whitelist/watchlist mechanism exists to bound the scan set.

Credentials in `config.yaml` are tracked by git with no `.gitignore` and no environment variable fallback.

## Trailing stop mechanics

The trailing stop logic in `manage_positions` updates `highest_price` if current price exceeds it, activates the trailing stop once price crosses `entry * (1 + activation)`, then sets `trailing_stop_price = highest_price * (1 - drop)`.

The stop price assignment happens unconditionally every tick once active. Because `highest_price` only ever increases, the computed stop also only increases. But the code doesn't express this invariant — it overwrites rather than guarding with `max()`. If a future change introduced a path where `highest_price` could decrease (e.g., position averaging), the stop would silently move down.

```python
# Current (fragile):
position.trailing_stop_price = position.highest_price * (1 - self.trailing_stop_drop)

# Safer:
new_stop = position.highest_price * (1 - self.trailing_stop_drop)
position.trailing_stop_price = max(position.trailing_stop_price or 0, new_stop)
```

The test suite doesn't cover a scenario where price oscillates above and below the activation threshold across multiple ticks to confirm the stop never decreases.

## Position lifecycle and crash recovery

Positions are held in `self.positions: Dict[str, Position]` with no persistence. The SQLite database records trade fills but not open position state. On crash, tokens purchased before the crash remain in the exchange wallet with no trailing stop or time stop monitoring them. The bot has no startup path that checks exchange balances or reconstructs positions from unpaired buy records in the database.

A minimal fix: persist positions to a `positions` table on buy, delete on sell, and load on startup. An alternative is querying the exchange wallet for non-zero balances and reconstructing entry metadata from the trades table.

## Blacklist desync between memory and database

Two independent systems track the same concept:

1. `BlacklistManager._loss_counts` — in-memory, maintained via `record_loss`/`record_win` calls during a single run
2. `Storage.get_consecutive_losses` — database query counting recent consecutive losing sells

`Storage.get_consecutive_losses` is never called anywhere in the codebase — it's dead code. After a restart, `_loss_counts` is empty, so a symbol that triggered the blacklist threshold immediately before shutdown trades freely after restart.

Fix: seed `BlacklistManager` from `Storage.get_consecutive_losses` at startup for each symbol that has recent trades, or remove the dead database method and document the in-memory limitation.

## Rate-limit exposure in the main loop

Each loop iteration calls `get_all_alpha_symbols()` (loads all markets) then `get_klines()` for every returned symbol. On mainnet with ~600 active USDT pairs, that's ~601 API calls per iteration. Binance's spot rate limit is 1200 weight/minute, and `fetch_ohlcv` carries weight 1-5. A single scan pass would take 30+ seconds with ccxt's rate limiting and still risks hitting weight-based caps.

The `time.sleep(5)` between iterations assumes scans complete quickly. With hundreds of symbols, the scan dominates loop time. The `filter_list` config is an exclusion list, not a whitelist — there's no way to bound the scan set to a manageable number of symbols.

## Credential handling

`config.yaml` is tracked by git (no `.gitignore` exists in the repo root). The `Exchange` constructor reads credentials exclusively from the config dict:

```python
api_key = binance_config.get("api_key", "")
secret = binance_config.get("secret", "")
```

No environment variable fallback. A developer who inserts real keys into the config and commits will leak credentials. Standard mitigation: `.gitignore` the config, ship `config.yaml.example`, and add `os.environ.get("BINANCE_API_KEY", ...)` priority.

## Test coverage gaps

Not tested:

- `Storage` — no integration tests for SQLite operations, schema creation, or duplicate trade ID handling.
- `Exchange` — no tests for retry logic, error classification, or the quantity calculation in `place_market_buy` (which divides `usdt_amount` by ticker price — a stale or zero price produces bad quantities).
- `FeishuNotifier` — no tests for retry behavior or the empty-webhook guard.
- `execute_sell` internals — profit/loss calculation, blacklist recording, and notification are not directly asserted; only `place_market_sell` is verified via mock.
- Tests use `datetime.now()` directly (not mocked), making cooldown and time-stop assertions timing-sensitive near midnight boundaries.
